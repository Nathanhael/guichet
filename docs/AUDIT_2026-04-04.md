## 3. Performance / Security

### No Rate Limiting on Push Routes (SECURITY)
`server/routes/push.ts` — both endpoints are unprotected:
- `POST /subscribe` — no auth, no rate limit
- `POST /unsubscribe` — no auth, no rate limit

**Risk**: Subscription flooding, resource exhaustion.
**Fix**: Add `auth` middleware + rate limiter (like `logoRateLimit` pattern).

### No Rate Limiting on Ticket Export (SECURITY)
`server/routes/tickets.ts:18` — `GET /export` has auth + role check but no rate limit.
**Risk**: Expensive query, potential DoS vector.
**Fix**: Add rate limiter (e.g., 5 exports/min/user).

### No ErrorBoundary in Any View (RELIABILITY)
All 5 views (`AdminView`, `AgentView`, `LoginView`, `PlatformView`, `SupportView`) lack `<ErrorBoundary>` wrappers. An uncaught render error crashes the entire app.
**Fix**: Wrap each view's return in `<ErrorBoundary>` with a recovery UI.

### Unhandled Promise Chains (RELIABILITY)
- `server/app.ts:76` — `initRedis().then(...)` — no `.catch()`
- `server/index.ts:13` — `bootstrapPlatformOperator().then(...)` — no `.catch()`
- `server/socket/handlers.ts:256,281` — fire-and-forget `.then()` without error handling

**Risk**: Silent failures during startup or socket handling.
**Fix**: Add `.catch(logger.error)` or convert to `await` in async context.

### AI `schema as any` Bypasses Tenant Isolation Checks (SECURITY)
The `as any` casts in the AI layer mean TypeScript cannot verify that partner-scoped queries include the `partnerId` filter. If a developer removes the filter by mistake, the compiler won't catch it.
**Fix**: Type the schema properly (see Tech Debt section above).

### SQL Patterns — OK
All raw SQL uses Drizzle's `sql` tagged template literal with parameterized values (`${cutoffDate}`, etc.). No injection risk detected.
